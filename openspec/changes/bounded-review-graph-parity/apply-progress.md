### UNIT 4 — RESL-001 (BLOCKER) — STOPPED, finding left open

Implemented exactly as instructed (`resolveRepositoryAuthorityV1` in `lib/review-repository.ts` re-keyed `repository_identity`/`repository_id`/`authority_id` to the canonical Git common-directory realpath instead of `git rev-list --max-parents=0 --all`), verified GREEN against a new RED test ("repository authority identity does not change when an unrelated orphan root commit appears", `tests/review-repository.test.ts`) and against updated versions of the two pre-existing tests that depended on root-commit-set identity (`tests/review-repository.test.ts` "non-Git directories and empty repositories fail closed" / `tests/review-transaction.test.ts` "repository authority fails closed until Git has stable root commit anchors").

**Discovered while running the full suite (not scoped to the 3 cited locations):** the fix breaks a separately and formally specified requirement — `openspec/changes/bounded-review-graph-parity/specs/review-graph/spec.md:151-165`, "Requirement: Validated atomic bundle transfer" / "Scenario: Valid export and import — … imported into a compatible repository" and "Scenario: Invalid bundle has no effect — … authority-forging … is rejected." `lib/review-bundle.ts:187` gates bundle import on exact `repository_identity`/`repository_id`/`authority_id` equality between the exporting and importing repository. Under root-commit-based identity, two independent clones of the same history share the same `repository_id` (portable) while each still has its own separate `store_root` (location-bound) — so a bundle exported from one clone imports cleanly into a genuinely different, empty clone (the documented "clean-clone/workstation recovery" path), while a truly unrelated repository is still rejected (`repository_id` differs). Common-directory-realpath identity is, by construction, never shared by two different directories, so after the fix **every** cross-clone import is rejected as "authority does not match" — confirmed by running `tests/review-bundle.test.ts` in isolation: 4 of 5 tests failed (`bundle export is deterministic and compatible import…`, `…reduced state does not match…`, `…claimed incarnation…`, `…malformed live STORE…`), all with `ReviewBundleError: Bundle repository authority does not match this repository` instead of their intended failure/success modes.

**Why this cannot be soundly resolved inside this correction transaction:**
- A test-only fixture change (using a linked Git worktree instead of `git clone` for the "compatible repository" fixture, so `target` shares `source`'s common directory) was tried and reverted: worktrees share not only `repository_id`/`authority_id` but also the literal `store_root` directory, collapsing "a distinct, initially-empty, compatible repository" into "the exact same store," which breaks the tests' load-bearing precondition that the target starts without the lineage before import (2 of 5 tests then failed for a new reason: `imported` was `false`/no exception was thrown, because there was nothing left to import — the state was already there).
- Preserving both properties (RESL-001's stability requirement, and the bundle spec's cross-clone-compatible-but-independently-empty requirement) needs a portable, content-derived compatibility signal that is deliberately *not* the store's own location — i.e., a second identity concept, decoupled from `resolveRepositoryAuthorityV1`'s location-bound authority, used only by `lib/review-bundle.ts`'s cross-repository check. ### Unit 2 — RISK2-001 (CRITICAL): STOPPED — finding left open, not faked

**Investigation performed:**

1. **Read the actual purpose of the gate's `common_directory` check** (`lib/review-transaction.ts:1532-1542` `assertCurrentRepositoryAuthority`, and `:1663-1666` `validateAuthoritativeGate`). In both cases, `common_directory` is compared between two authority resolutions **for the same `cwd`, in the same process, at two different times** (store-construction time vs. gate-validation time). Its purpose is detecting that the *live* authority for one repository has drifted since the store/receipt was created (e.g. a symlink swap, or the repo being torn down and recreated) — a same-repository staleness/tamper check. It is never used to compare two *different* physical repositories' `common_directory` values against each other.
2. **Read the actual bundle-import code path** (`lib/review-bundle.ts:184-187`): the ONLY check today is `repository_identity`/`repository_id`/`authority_id` equality between the bundle manifest and `this.#authority` — all three are hashes derived purely from the pinned root-commit set (never from `common_directory`).
3. **Read the reference model** at `/home/gentleman/work/gentle-ai/internal/reviewtransaction/bundle.go` (read-only, as instructed). Its `ImportBundle`/`validateBundleExpectation` do not carry or compare any `repository_identity`/`repository_id`/`common_directory`-equivalent field at all. Its `AuthoritativeStore` (`store.go:46-73`) derives the store path **directly from the physical Git common directory of the repo the call is running against** — there is no abstract identity blob to forge in the first place. Its actual anti-forgery binding is instead literal content equality: `expected.Snapshot.CandidateTree`, `PathsDigest`, and a re-derived `IntendedUntrackedProof` must match the **current, live candidate tree of the destination repository** (`bundle.go:222-228`). This is a fundamentally different architecture (physical-path-derived store + live-content binding) than gentle-pi's `review-bundle.ts` (abstract graph-object-store transfer keyed by a portable root-commit-hash identity) — there is no "correct bundle-bound field" in the Go reference to port over; the two designs solve the cross-repository trust problem by different means, and Go's approach doesn't need a common_directory field because it never abstracts identity away from the physical repository at all.
4. **Empirically verified `common_directory` is clone-specific**, not a shared/portable value, using the exact scenario `tests/review-bundle.test.ts`'s `repository()` helper builds (source repo + `git clone` target): `git -C source rev-parse --path-format=absolute --git-common-dir` → `/tmp/.../source/.git`; `git -C target rev-parse --path-format=absolute --git-common-dir` → `/tmp/.../target/.git` — different absolute paths — while `git rev-list --max-parents=0 --all` returns the **identical** root-commit hash in both (clones share history). This confirms: adding a literal `manifest.common_directory !== this.#authority.common_directory` equality check to `ReviewBundleImporter.import` would make the "bundle export is deterministic and compatible import installs its exact graph closure atomically" test (and every other cross-clone scenario in the 6-test `review-bundle.test.ts` suite, all of which use `source`/`target` as separate clones) fail immediately — a real regression, not a false alarm.

**Why the finding cannot be soundly closed within this unit's scope:**

- The forgery scenario is real (an attacker who has read access to a public repo's root commit object(s) — or a leaked bundle manifest revealing `repository_identity` — can graft just those commit(s) as an orphan root in an unrelated scratch repo and obtain a colliding `repository_id`/`authority_id`, since `resolveRepositoryAuthorityV1` only requires exact match of the pinned root-commit *set*, not full shared history).
- But the only concrete field this codebase uses to distinguish "the same physical repository" from "a different one" (`common_directory`) is **not stable across legitimate clones**, which is the exact scenario bundle export/import exists to support. Enforcing it would defeat the feature's own purpose and break already-passing, intentional tests.
- The reference model's actual defense (live content-tree binding) is not a small, bounded, drop-in addition to `ReviewBundleImporter.import` — `review-bundle.ts`'s bundle format has no concept of "candidate tree" or "current live review transaction" to bind against; it transfers graph objects keyed by abstract lineage/authority identity. Building an equivalent binding would require designing a new cross-repository trust primitive (e.g. bundle signing, an explicit trust list, or extending pinned identity to a broader, non-`O(root-count)` portion of the commit graph) — a genuine architectural change, not a correction-unit-sized fix, and out of scope for "fix exactly the 3 corroborated findings... no unrelated refactors."

No code was changed for this unit. RISK2-001 remains open and should be routed back for an explicit design decision (maintainer call on which trust primitive to add) rather than closed by a mechanical patch that would either break portability or provide no real security improvement.

## Deferral: graph-v1 cross-repo bundle trust (RISK2-001) — experimental, deferred

**Maintainer decision:** ship the parity-that-matters (release-from-main fast path, canonical hashing, lock, receipts, gates, pinned-subset identity — done and green) and defer closing `RISK2-001` to its own follow-up change, `openspec/changes/cross-repo-bundle-trust`, rather than block the release on an architectural cross-repo trust primitive.

**Why deferred, restated:** Unit 2 above already establishes, with direct investigation of `lib/review-transaction.ts`'s `common_directory` gate, `lib/review-bundle.ts`'s import path, and the `gentle-ai` Go reference model (`internal/reviewtransaction/bundle.go`, read-only), that `common_directory` cannot discriminate a legitimate same-repository clone from a forged foreign identity — it legitimately differs across clones of the exact same repository, which is precisely the scenario bundle export/import exists to support. The Go reference solves this with a structurally different, physical-path-plus-live-content-tree binding that has no equivalent field in `review-bundle.ts`'s abstract graph-object-store transfer model. Closing `RISK2-001` for real requires a genuine new cross-repo trust primitive (signed manifest, content-tree binding, or trust-on-first-use pinning), not a bounded correction — hence the separate follow-up change rather than reopening this terminal, escalated review lineage.

**Guard added so the gap is non-silent:** `ReviewBundleImporter.import` (`lib/review-bundle.ts`) now takes an experimental `acknowledgeUntrustedBundleSource?: boolean` import option (default false/absent). Adopting any lineage this store's own authority has not already established fails closed with a distinct `REVIEW_BUNDLE_UNTRUSTED_SOURCE` error unless the caller explicitly passes `acknowledgeUntrustedBundleSource: true` — an experimental, operator-attested trust decision pending the follow-up primitive. Re-importing or extending an already-locally-established lineage is unaffected and needs no acknowledgement. `extensions/gentle-ai.ts`'s `import` controller operation was wired to accept and forward this same flag (`acknowledgeUntrustedBundleSource: "true"` string parameter) so the real CLI/tool surface stays usable, not just the unit tests.

**Test evidence:** `tests/review-bundle.test.ts` — the original 6 tests were updated to pass `acknowledgeUntrustedBundleSource: true` on the import calls that adopt a lineage for the first time (re-imports of an already-known lineage are unaffected and still pass no flag); 3 new tests added: import of a brand-new lineage denied without the acknowledgement (`REVIEW_BUNDLE_UNTRUSTED_SOURCE`), import of a foreign clone's independently-fabricated lineage denied the same way despite a matching `repository_identity` (directly demonstrating the RISK2-001 vector is symmetric — a legitimate bundle and a foreign one are indistinguishable by identity alone), and import allowed once the caller explicitly acknowledges the untrusted source. Full suite: `pnpm test` — PASS (417/417; the 414 baseline plus this round's 3 new tests; runtime harness PASS).

**What ships vs. what's deferred:** same-repo/same-clone portability ships and works (with the caller passing the explicit opt-in for first-time lineage adoption); untrusted cross-repo import of a brand-new lineage requires explicit operator attestation pending the follow-up trust primitive designed in `openspec/changes/cross-repo-bundle-trust`.
